# In-Field Test (IFT) Firmware Design

## Overview

The In-Field Test (IFT) subsystem provides automated memory and core testing capabilities for Kingsgate firmware. It supports asynchronous test execution, dual-die coordination, and reporting of test results to the HSP and BMC. IFT is tightly integrated with the Driver Framework (DFWK) and ICC mailbox communication.

---

## Architecture

IFT consists of several components:
- **IFT API Layer** (`ift.c`, `ift_fw.h`): Public APIs for test setup, execution, and status queries.
- **Driver Framework Integration** (`ift_dfwk.c`): Handles async request dispatch, device/interface initialization, and completion routines.
- **ICC Mailbox Layer** (`ift_icc.c`): Manages communication with HSP and die-to-die coordination.
- **Private/Internal Logic** (`ift_i.h`): Internal helpers, constants, and private APIs.

## Driver Framework Integration

IFT binds into DFWK through the helper layer in [src/libs/ift/src/ift_dfwk.c](../../../src/libs/ift/src/ift_dfwk.c). A static interface pointer is populated by `ift_dfwk_set_interface` during bring-up and is reused whenever requests are issued. The flow is split across three responsibilities:

- **Device bring-up** (`ift_dfwk_device_initialize`): Calls `DfwkDeviceInitialize` for the IFT device header and wires a serialized queue via `DfwkQueueInitialize`, pointing dispatch work to `ift_dfwk_dispatch`.
- **Interface bring-up** (`ift_dfwk_interface_initialize`): Stores the owning device, initializes the interface with the device queue through `DfwkInterfaceInitialize`, and immediately opens it with `DfwkClientInterfaceOpen` so asynchronous traffic can be posted.
- **Async request path** (`ift_dfwk_send_async_req`): Programs the request type, runs `DfwkAsyncRequestInitialize`, attaches a completion callback (`ift_dfwk_test_complete_cb`), and pushes the request to DFWK via `DfwkInterfaceSendAsync`.

All requests funnel through `ift_dfwk_dispatch`, which logs the incoming type and executes the relevant action:

```c
switch (request->RequestType)
{
case IFT_MEM_TESTS_FW_LOAD_ASYNC:
case IFT_CORE_TESTS_FW_LOAD_ASYNC:
    ift_load_fw_sos(request);
    break;
case IFT_EXECUTE_FW_ASYNC:
    ift_execute_test(request);
    break;
default:
    DfwkAsyncRequestComplete(request);
    break;
}
```

The completion handler currently discards its parameters and serves as a synchronization hook for future extensions.

## ICC Mailbox Communication

IFT relies on the ICC helper layer in [src/libs/ift/src/ift_icc.c](../../../src/libs/ift/src/ift_icc.c) to talk to both HSP and the peer SCP die. The interactions break down into three flows that are orchestrated from [src/libs/ift/src/ift.c](../../../src/libs/ift/src/ift.c).

- **Intent discovery**: `ift_init` calls `ift_icc_get_ift_intent` to synchronously fetch the enable flag and intent type. The request validates the mailbox response headers before caching the values that drive the remainder of the sequence.
- **Die-to-die firmware transfer**: After copying binaries to die 1 MSCP EXP RAM, die 0 sends `ift_icc_send_d2d_fw_done_msg`. Die 1 enables the matching receive path via `ift_icc_register_d2d_fw_done`, whose callback updates the current firmware size and schedules execution on the local DFWK queue.
- **Result streaming and status**: `ift_icc_send_test_result_and_status` walks the result buffer one 32-bit word at a time. It uses a static index to chain asynchronous `fpfw_icc_base_send_recv` calls through `ift_icc_recv_results_cb`; once all entries are drained it flips over to `ift_icc_send_test_status`, which packages both the silibs execution status and mismatch flag. The status callback `ift_icc_recv_status_cb` notifies die 0 completion when running on die 1 so system reset does not occur prematurely.

Throughout the exchange, assertions guard mailbox contract violations and completion callbacks keep the request pipeline active without blocking the calling context.

## IFT environment setup

When intent enables IFT, firmware pivots into a minimized runtime profile that keeps HSP communication online while disabling background services that could interfere with deterministic test execution. The setup surfaces in multiple modules:

- **IFT runtime bring-up**: `ift_init` saves the ICC context, reprograms the MSCP EXP SRAM region as strongly ordered, and refreshes caches before any binaries run ([src/libs/ift/src/ift.c](../../../src/libs/ift/src/ift.c#L240-L309)). The paired init component also records the HSP mailbox IRQ number to keep it enabled and disables rest of the interrupts ([src/libs/ift/src/ift.c ift_disable_irq()](../../../src/libs/ift/src/ift.c#L114)).
- **Clocking tweaks**: CSS post-mesh init flips `ift_mode_en` so PLL and counter sequencing match the reduced-service test mode ([src/libs/css/css.c](../../../src/libs/css/css.c#L90-L103)).
- **Service suppression**: Boot-complete notification, Bitrot scrubbing, DDR manager threads, and PCIe configuration workers either short-circuit or avoid spawning entirely when IFT is active, preventing them from touching memory or firmware channels reserved for tests ([src/libs/runtime_init/accel_boot_notify_init.c](../../../src/libs/runtime_init/accel_boot_notify_init.c#L24-L54), [src/libs/runtime_init/bitrot_service_init.c](../../../src/libs/runtime_init/bitrot_service_init.c#L36-L79), [src/libs/ddrss/ddr_manager/ddr_manager_top.c](../../../src/libs/ddrss/ddr_manager/ddr_manager_top.c#L60-L122), [src/services/pcie_manager/pcie_config_manager_thread.c](../../../src/services/pcie_manager/pcie_config_manager_thread.c#L150-L210), [src/services/pcie_manager/pcie_rpss_manager_thread.c](../../../src/services/pcie_manager/pcie_rpss_manager_thread.c#L120-L230)).
- **PCIe training adjustments**: RPSS threads bypass link training and PHY bring-up sequences, issuing only the minimal configuration commands required by downstream components ([src/services/pcie_manager/pcie_rpss_manager_thread.c](../../../src/services/pcie_manager/pcie_rpss_manager_thread.c#L180-L260)).
- **Power management overrides**: Core power init treats every fused-out core as enabled, forces DVFS software boot mode, and relies on direct DVFS/ODCM programming instead of PPU handshakes. AP core PPU helpers likewise skip power-state transitions so IFT can manipulate cores without waiting on firmware-managed power domains ([src/services/power/src/power_hw_int.c](../../../src/services/power/src/power_hw_int.c#L740-L1030), [src/services/ap_core/ap_core_ppu.c](../../../src/services/ap_core/ap_core_ppu.c#L185-L220)).

Collectively these changes ensure IFT owns the execution environment: tests run with deterministic memory attributes, mailbox access remains uninterrupted, and subsystems that might race the test binaries stay idle.

## Error Handling and Assertions

IFT uses `BUG_ASSERT` and `BUG_ASSERT_PARAM` macros to validate parameters and state throughout the code. All mailbox and memory operations are checked for success.

---

## Extensibility

IFT is designed to support additional test types and firmware binaries. The driver framework and mailbox layers allow for easy integration of new test flows and reporting mechanisms.
